Unopinionated mechanism to load i18n content

We can't embed CLDR data into our code again, or we'll keep maintaining it as we've been before..

We can't simply switch to CLDR without redesigning the way data is loaded

We believe libraries should not mandate on how users load this in their app. It's like: You should do it in your own way.

Users have different needs, and they have different preferences.

Also...

Avoid duplicating data across multiple i18n CLDR libraries

Also, by avoiding to embed content, we can share the same data across multiple libraries.

That means: in your app, you load data once, and you use the same data for every different i18n library you use.

Use the load method and pass the JSON content. So, you are free to do it the way you want.

Note that you can pass this JSON the way you want. Also note that you don't need to load the entire CLDR database in your application. Only load the content and locales you need.

Now, this is important. Although you CAN use the vanilla data from CLDR servers, you don't NEED to stick to it.

Let's say you tweaked your locale data, or you simply don't want to wait for something to be fixed upstream.

Feel free to fix it locally in your application and have it working right ahead.

CLDR is a pretty complex and extensive database. It has so many rules, multiple inheritance, and so many things

To abstract part of that work away from Globalize, we created cldr.js. So, we keep Globalize focused on the i18n functions only.

cldr.js

Simple library to facilitate I18n software to access and use the official CLDR JSON data.

https://github.com/rxaviers/cldr.git

Cldr.js is a low level library, whose only purpose is to help to manipulate CLDR.

So, it's really cool to develop Globalize not needing to worry about that.
